# Checklist — Adding a New Boss

Turnkey numbered checklist for adding a stage boss (or mid-boss, scaled down) end to end.

1. **Create the `BossData` resource** (`.tres`) — id, display name, weakness archetype (one of Frost/Flame/Storm/Volt/Stone/Toxin/Blade/Gravity), HP, contact-damage value, reward weapon reference, phase/attack table (below), and the WorldFlag key this boss sets on defeat.
2. **Build the arena scene.** A boss arena is typically a single-screen or near-single-screen space with no scrolling mid-fight (or deliberately limited scrolling if the design calls for an arena larger than one screen) — confirm camera bounds are locked to the arena, not inherited from the stage's traversal camera settings.
3. **Build the intro sequence.** Door-seal or entrance beat, camera pan/focus to the boss, boss-intro sting jingle (see `assets/audio-spec.md`), then control handed back to the player as the fight starts. This must be skippable or fast on retry (see `gotchas.md` #11 — do not force a full unskippable intro replay on every retry after a death, or checkpoint-adjacent retries become tedious).
4. **Build the phase/attack table**, with an explicit telegraph for every attack:

   | Attack | Telegraph | Damage/effect | Notes |
   |---|---|---|---|
   | (per attack) | Describe the visible/audible tell and its lead time | | Every entry must have a telegraph — see gate 3 below. |

   Structure phases by HP threshold (e.g., phase 2 unlocks new attacks or changes attack frequency below 50% HP) rather than by timer, so player skill (doing more weakness damage) is what advances the fight, consistent with the reward structure of finishing bosses faster via correct weapon choice.
5. **Wire the weakness reaction and stagger.** When hit by this boss's assigned weakness weapon (from the cycle — see `checklists/new-weapon.md`), the boss must take bonus damage (typically 2-3x normal weapon damage) **and** play a distinct hit-reaction/stagger animation+SFX so the weakness hit is unmistakable even without checking a health bar. A weakness hit that looks identical to a normal hit is a design failure — the whole cycle mechanic depends on this feedback being obvious.
6. **Wire the reward weapon grant.** On defeat, trigger the get-weapon fanfare jingle (see `assets/audio-spec.md`) and grant the reward `WeaponData` to the player's inventory — confirm this happens through the same weapon-unlock path used everywhere else (see `checklists/new-weapon.md`), not a one-off special-cased unlock just for this boss.
7. **Set WorldFlags on defeat.** Exactly one flag write marking this boss defeated, performed as part of the same save transaction as any other end-of-fight state change (see `gotchas.md` #11 — single save transaction after boss defeat, not scattered writes across the death/reward/transition sequence).
8. **Re-check weakness-chart integrity.** Confirm this boss's assigned weakness slots into the 8-archetype cycle with no duplicate assignment and no self-loop (a boss must never be weak to the weapon it itself grants) — see the full cycle-integrity rule in `checklists/new-weapon.md`. Run this check again here even if it was already checked when the corresponding stage/weapon was added — a boss is the point where a chart error actually becomes unbeatable-feeling rather than just a data mismatch.
9. **Playtest gate — buster-only beatable, ~2-3 minutes.** A player using only the starting buster (no elemental weapons) must be able to beat this boss, and a competent-but-not-perfect playthrough should land in roughly the 2-3 minute range. Significantly faster suggests the boss is undertuned for a "first encounter" fight; significantly slower (or not beatable at all) is a hard blocker — buster-only must always be a valid, if harder, path.
10. **Playtest gate — weakness kill under 1 minute.** A player using the correct weakness weapon at reasonable skill should be able to beat this boss in under a minute. This large gap between buster-only and weakness-optimal time is the core reward loop for the weapon-cycle design — if the weakness weapon doesn't produce a dramatically faster kill, its damage multiplier (step 5) is too low.
11. **Playtest gate — every attack reactively dodgeable.** For each row in the phase/attack table, confirm a player seeing the telegraph for the first time (not memorizing patterns in advance) has enough lead time to react and avoid or mitigate the attack. An attack with no fair telegraph, or a telegraph too short to react to, must be redesigned before this boss is considered content-complete — memorization-only bosses break the "readable, fair" contract this genre depends on.
