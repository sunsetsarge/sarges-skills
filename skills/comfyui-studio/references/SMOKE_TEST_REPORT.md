## Rules from failures

1. **`LoraLoaderModelOnly`'s dropdown (via `get_node_info`/`validate_workflow`) reflects a startup-time cached model list, not the live directory.** After downloading a new LoRA mid-session with `download_model`, `list_local_models` (live REST/filesystem query) sees the file immediately, but `validate_workflow` / `get_node_info` reported `value_not_in_list` for the exact same filename. Fix: **skip `validate_workflow` for freshly-downloaded files and go straight to `enqueue_workflow`** â€” the actual ComfyUI server-side execution path resolves against the live `models/loras/` directory correctly and the job runs fine. Restarting ComfyUI would also refresh the cache but is disruptive mid-session.

2. **"0%, Model Initializing" on GGUF UNET loads with heavy CPU offload (5-5.7GB offloaded on this 10GB card) can legitimately take 15 seconds to 90+ seconds per step before the tqdm bar visibly advances** â€” this is NOT a hang. Root cause: first-time CUDA kernel JIT compilation for the GGUF dequantization ops, combined with genuine PCIe-bound weight streaming for the ~5GB of UNET layers that don't fit in the remaining VRAM after the 6.2GB Qwen2.5-VL-7B CLIP is loaded. Confirmed via one interrupted job's own final log line: `25%| 1/4 [01:28<04:24, 88.32s/it, Model Initialization complete!]` â€” i.e. it WAS making progress, at ~88s/step, right up until the interrupt landed. **Fix: do not cancel a GGUF job showing 0% before at least 3-5 minutes of wall time have passed**, especially when `get_system_stats` shows falling `ram_free` (evidence of active paging, not a stall). This cost 4 needlessly-cancelled T2 attempts before the pattern was recognized (T2's 5th attempt at 512x512/4-step succeeded once left alone).

3. **Confirmed server crash: ComfyUI Desktop became completely unreachable (`fetch failed` / `CONNECTION_ERROR` on every subsequent MCP call, including `health_check`) partway through the T3 edit run**, after repeated GGUF UNET loads/unloads across T2 and T3 attempts progressively consumed system RAM (`ram_free` fell from 27.6GB at session start to ~9-12GB by the time of the crash) on top of the already-tight 10GB VRAM. `restart_comfyui` reported *"Could not stop ComfyUI: No process found listening on port 8000"* (process had already died) and `start_comfyui` reported *"No previous process info and could not find ComfyUI Desktop app. Start ComfyUI manually."* â€” **the MCP server cannot auto-relaunch ComfyUI Desktop; this requires manual relaunch by the user.** Likely root cause: cumulative RAM pressure from repeated large-model (6-7GB CLIP + 5-7GB offloaded UNET) load/unload cycles without full release between cancelled jobs, eventually exhausting available system memory. **Recommendation for the next session: (a) relaunch ComfyUI Desktop, (b) run `clear_vram` + verify `get_system_stats` shows `ram_free` back near the ~27GB baseline before starting T3 again, (c) consider adding `--reserve-vram 2` or `--novram` launch flags for the Qwen GGUF family specifically per the troubleshooting skill's VRAM ladder, since 10GB is tight for a 13GB-class quantized model plus a 6-7GB VL text encoder simultaneously.**

4. **`qweneditutils` custom node pack (which provides `TextEncodeQwenImageEditPlusAdvance_lrzjason`, the skill's recommended node) is not installed on this server** â€” `get_node_info` returned "No nodes found matching" for it. Used the documented fallback instead: built-in `TextEncodeQwenImageEditPlus` (confirmed present, image1/vae/prompt/clip inputs) plus a separate `EmptyLatentImage` node, per the qwen-image-edit skill's "Simpler Alternative (Phr00t v2)" section. This fallback graph was later proven end-to-end in the post-relaunch T3 run (PASS, `studio_t3_00001_.png`) and is what the saved `studio-edit-qwen2511.json` uses.

5. **`Qwen2.5-VL-7B-Instruct-mmproj-BF16.gguf` auto-attaches automatically** â€” the `ComfyUI-GGUF` `CLIPLoaderGGUF` node logs `"Attenpting to find mmproj file for text encoder... Using mmproj 'Qwen2.5-VL-7B-Instruct-mmproj-BF16.gguf' for text encoder 'Qwen2.5-VL-7B-Instruct-UD-Q4_K_XL.gguf'."` with no explicit wiring needed â€” the vision tower is found and loaded by convention/filename-matching alongside the base GGUF text encoder. No manual mmproj slot exists on `CLIPLoaderGGUF`; this resolves the task brief's anticipated "missing mmproj" contingency â€” it was never actually missing.

6. **ComfyUI Desktop can crash outright under system-RAM pressure from repeated GGUF family switches.** Repeated large-GGUF load/unload cycles (6-7GB VL text encoder + 5-7GB offloaded UNET per cycle) drained `ram_free` from 27.6GB to ~9GB and then killed the server process entirely mid-run. Avoid rapid load/unload churn: **one clean attempt per model family, sequential**, with a single `clear_vram` between families â€” not between retries of the same family.

7. **After a crash, relaunching ComfyUI Desktop can bind port 8188 instead of 8000** (stale TIME_WAIT on 8000, and the Desktop app's bundled fallback backend defaults to 8188). Confirmed twice this session: plain `Start-Process ComfyUI.exe` produced a backend on 8188 running an OLD bundled ComfyUI 0.22.3, while the MCP target is pinned to 8000. Remedy: kill the app AND any python on 8188/8000, wait ~10-15s, relaunch, and verify `/system_stats` responds **on 8000 with the expected version (0.27.0)** before enqueueing anything. In this session the reliable recovery was launching the real server directly: `Start-Process "C:\ComfyUI\.venv\Scripts\python.exe" -ArgumentList "C:\Users\blain\ComfyUI-Installs\ComfyUI\ComfyUI\main.py","--port","8000","--base-directory","C:/ComfyUI"`.

8. **The MCP `restart_comfyui`/`start_comfyui` tools cannot restart the Desktop app** (`spawn EFTYPE` / "could not find ComfyUI Desktop app"); `restart_comfyui` may even kill the server and then fail to respawn it, leaving nothing on 8000. Recover via PowerShell per Rule 7. The Desktop app lives at `C:\Users\blain\AppData\Local\Programs\@comfyorgcomfyui-electron\ComfyUI.exe`, but note Rule 7 â€” launching it plainly gives you the wrong (bundled 0.22.3, port-8188) backend.

9. **Version-mismatch after relaunch: `AttributeError: 'ModelMMAP' object has no attribute 'get_file_handle'` on EVERY safetensors load (VAELoader failed in 10-21ms).** ComfyUI 0.27.0 core (`comfy/utils.py` `load_safetensors`) requires `comfy-aimdo==0.4.10`, but the relaunched server booted with `comfy-aimdo 0.3.0` (visible in `get_system_stats` â†’ `comfy_package_versions`: installed vs required). Worse, the venv had a corrupt duplicate dist-info ("Ignoring invalid distribution ~ip" warnings): `pip show` claimed 0.4.10 after install while `importlib.metadata`/actual import still resolved 0.3.0. Fix: `pip uninstall comfy-aimdo -y` (repeat until clean), then `pip install comfy-aimdo==0.4.10 --force-reinstall --no-deps` into `C:\ComfyUI\.venv`, verify with `python -c "import comfy_aimdo.model_mmap as m; print(hasattr(m.ModelMMAP,'get_file_handle'))"`, then restart the server. **Always check `comfy_package_versions` installed-vs-required in `get_system_stats` after any relaunch.**

10. **Cold-cache GGUF runs are 5-20x slower than warm runs.** T3's successful post-crash run took 22.4 min (init 6:09 + ~310-350s/step) because the 12GB edit GGUF had to page in from disk on a fresh boot with only ~4.5GB usable VRAM headroom; T2's warm-cache run of a same-size model was 62s. Budget accordingly: first GGUF job after a server restart may take 15-25 min at 4 steps. tqdm progress lines only flush to `get_logs` on step COMPLETION, so a silent log with `running: true` and shifting VRAM/RAM numbers means work, not a hang (extends Rule 2: with a cold cache the no-cancel window is 10+ minutes, not 3-5).
